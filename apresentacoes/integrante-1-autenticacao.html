Autenticação & Controle de Acesso

Integrante 1 — como o usuário entra no sistema e como o acesso é protegido

Antes de começar

Prepare o ambiente para conseguir mostrar o código e o sistema rodando.

  • Abra a pasta do projeto no VS Code.
  • No terminal, rode npm install (só na primeira vez) e depois npm run dev.
  • Abra http://localhost:5173 no navegador — vai cair na tela de login, que é justamente a sua parte.
  • Login de demonstração: admin@ama.org / admin123.
  • Os seus arquivos ficam em src/. Deixe-os abertos em abas para alternar durante a fala.
src/App.tsx src/contexts/AuthContext.tsx src/components/PrivateRoute.tsx src/pages/Login.tsx src/pages/Register.tsx src/types/user.ts

Seu módulo em uma frase

“Eu cuidei de tudo que envolve entrar no sistema: as telas de login e cadastro, o controle de quem está logado e a proteção que impede um usuário não autenticado de acessar o sistema.”

Roteiro passo a passo

Siga nesta ordem — vai do “esqueleto” (rotas) para o “miolo” (contexto) e termina na tela.

1

O esqueleto das rotas

Abra: src/App.tsx

Mostre o componente App. Aponte para o <BrowserRouter> e as três rotas: /login, /register e / (a página principal).

Destaque que a rota / está “embrulhada” por <PrivateRoute> — é isso que a protege. Mostre também que tudo está dentro de <AuthProvider> e <ToastProvider>.

“A aplicação tem três rotas. A rota raiz é protegida: ela só renderiza a página principal se o usuário estiver logado — quem garante isso é o PrivateRoute. E tudo fica dentro do AuthProvider, que é o cérebro da autenticação.”
2

O cérebro: o contexto de autenticação

Abra: src/contexts/AuthContext.tsx

Esta é a parte mais importante de explicar. Mostre o estado token, que é inicializado lendo o localStorage — por isso o usuário continua logado mesmo após atualizar a página.

Aponte para as funções login() e logout(): login salva o token e registra a entrada no histórico (logLogin); logout registra a saída e apaga o token.

Mostre a linha role = token ? getUserRole(token) : null e o isAdmin — é daqui que sai a informação de quem é administrador. Por fim, o hook useAuth(), que os outros componentes usam para acessar tudo isso.

“Em vez de passar o login de componente em componente, eu centralizei isso num Context do React. Qualquer parte do sistema chama useAuth() e já sabe quem está logado, qual o papel da pessoa e se ela é admin.”
3

O porteiro: proteção de rota

Abra: src/components/PrivateRoute.tsx

É um arquivo curtinho, fácil de explicar. A lógica é uma linha: se existe token, renderiza o conteúdo; senão, <Navigate to="/login"> redireciona para o login.

“Esse componente é o porteiro. Se você tentar acessar a página principal sem estar logado, ele te joga de volta pra tela de login automaticamente.”
4

A tela de login

Abra: src/pages/Login.tsx · e mostre no navegador

Mostre a função handleSubmit: ela chama loginUser(email, senha) para validar as credenciais e, se der certo, chama o login(email) do contexto e navega para /.

No navegador, mostre o botão de mostrar/ocultar senha (o ícone de olho — estados showPassword com Eye/EyeOff) e a mensagem de erro quando a senha está errada.

Comente que a tela de cadastro (Register.tsx) segue o mesmo padrão, chamando registerUser.

“Quando o usuário envia o formulário, eu valido o email e a senha. Se bater, salvo a sessão pelo contexto e mando pra home. Se não bater, mostro o aviso de erro. Coloquei também o botão de olho pra revelar a senha, melhorando a usabilidade.”
5

Os tipos (papéis)

Abra: src/types/user.ts

Mostre UserRole = "admin" | "voluntario" e o ROLE_LABELS (que traduz o papel para exibição). Diga que esses tipos são usados por todo o sistema para decidir permissões.

Como seu módulo conversa com o resto

  • O token salvo é o próprio e-mail do usuário. Isso é usado pelo módulo de Histórico para saber “quem fez” cada ação.
  • A página principal (Home.tsx) usa o seu useAuth() para o botão Sair, para mostrar o papel e para decidir se a aba “Usuários” aparece (só para admins).
  • O seu login() e logout() disparam os registros de entrada/saída no Histórico (Integrante 5).
  • As funções loginUser / registerUser que você chama ficam na camada de dados storage.ts (Integrante 9).

Sugestão de fala ~4 min

  • 0:00–0:30 — “Minha parte é a porta de entrada do sistema.” Mostre a tela de login rodando.
  • 0:30–1:30 — Abra App.tsx: rotas + a ideia de rota protegida.
  • 1:30–2:45 — Abra AuthContext.tsx: token no localStorage, login/logout, papel e useAuth.
  • 2:45–3:15 — PrivateRoute.tsx: o porteiro.
  • 3:15–4:00 — De volta ao navegador: faça login ao vivo, mostre o botão de olho e o erro de senha.

Perguntas que o professor pode fazer

Por que usar Context em vez de passar props?
Porque a informação de login é usada em muitos lugares (header, abas, histórico). O Context evita “prop drilling” — passar a mesma prop por vários níveis — e centraliza a lógica num só lugar.
Por que o usuário continua logado após atualizar a página?
Porque o token é guardado no localStorage e o contexto lê esse valor ao iniciar. Ao recarregar, ele recupera a sessão.
Como o sistema sabe se alguém é administrador?
O contexto pega o papel do usuário com getUserRole(token) e expõe isAdmin. A interface usa isso para liberar ou esconder áreas restritas.
Isso é seguro para produção?
Não — e isso é proposital, é um projeto acadêmico. As senhas ficam em texto no localStorage e não há backend. Num sistema real usaríamos servidor, senha com hash e tokens assinados (JWT).
