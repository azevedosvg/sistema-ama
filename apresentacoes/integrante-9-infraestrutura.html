Integrante 9 — onde os dados vivem e como o projeto roda

Antes de começar

  • Sua parte é a “fundação” — não tem uma tela só sua, mas todo o sistema depende dela. Sua estrela é o arquivo storage.ts.
  • Rode npm run dev. Para mostrar a persistência: cadastre um produto, atualize a página (F5) e mostre que ele continua lá.
  • Dica visual: abra as Ferramentas do Desenvolvedor (F12) → aba Application → Local Storage. Você vai ver as chaves ama_products, ama_users etc. — é o seu “banco de dados”.

Roteiro passo a passo

1

O ponto de partida da aplicação

Abra: src/main.tsx

Arquivo curtinho: mostre o createRoot(...).render(<App />) — é onde o React “liga” e injeta a aplicação dentro do index.html. Cite o vite.config.ts como a configuração que monta e serve o projeto.

“Tudo começa aqui: o main.tsx é o ponto de entrada que injeta a aplicação React na página. O Vite é a ferramenta que compila e serve isso durante o desenvolvimento.”
2

O “banco de dados”: as chaves do localStorage

Abra: src/lib/storage.ts → const KEYS

Mostre o objeto KEYS: produtos, usuários, transações, movimentações, atividades. Explique que toda a persistência é feita no localStorage do navegador — não há servidor nem banco externo.

“Em vez de um banco de dados tradicional, eu uso o localStorage do navegador como armazenamento. Cada tipo de dado tem a sua chave. É simples, não precisa de servidor, e os dados sobrevivem ao recarregar a página.”
3

O padrão de leitura/escrita (CRUD)

Abra: src/lib/storage.ts → readRaw, writeRaw, nextId, createProduct...

Mostre o padrão que se repete para cada entidade: uma função para ler, uma para gravar e um gerador de IDs incrementais. Mostre createProduct como exemplo: gera o ID, grava e registra a atividade. Diga que produtos, transações, movimentações e usuários seguem o mesmo padrão.

“Eu criei um padrão consistente: ler, escrever e gerar IDs. Cada operação de criar, editar ou excluir também chama o registro de atividade — por isso a auditoria funciona pra tudo automaticamente.”
4

Dados de exemplo e migração (a parte esperta)

Abra: src/lib/storage.ts → SEED_VERSION, migrateSeedIfNeeded, PRODUCT_SEED, TRANSACTION_SEED

Mostre os seeds — 47 produtos e 32 transações que já vêm preenchidos, para o sistema nunca abrir vazio. Depois explique migrateSeedIfNeeded: quando a SEED_VERSION muda, ele recarrega os dados de exemplo preservando as contas de usuário e o histórico.

“Pra demonstração não começar com tela vazia, eu deixei dados de exemplo prontos. E criei um controle de versão do seed: se eu atualizo os dados de exemplo, o sistema recarrega só eles, sem apagar os usuários nem o histórico que já existiam.”

Como seu módulo conversa com o resto

  • É o ponto central: todos os hooks (produtos, finanças, movimentação, histórico, usuários) importam funções daqui.
  • O seu logActivity dentro de cada operação é o que faz a auditoria (Integrante 5) funcionar automaticamente.
  • Guarda o token de login (Integrante 1) e fornece os dados de produtos que o estoque (Int. 2) e a movimentação (Int. 4) manipulam.
  • A configuração (Vite, TypeScript, ESLint) é o que permite todos rodarem e construírem o projeto.

Perguntas que o professor pode fazer

Por que usar localStorage em vez de um banco de dados?
Por simplicidade do projeto: não precisa de servidor nem instalação. Os dados ficam no navegador e persistem ao recarregar. A desvantagem é que ficam só naquele navegador — num sistema real usaríamos um backend com banco.
Como os IDs são gerados?
Com um contador incremental guardado no próprio localStorage (ex.: ama_next_id): a cada novo item, leio o valor, uso e incremento.
Para que serve o SEED_VERSION e a migração?
Para atualizar os dados de exemplo sem perder o que importa. Se a versão muda, recarrego produtos e transações de exemplo, mas preservo as contas de usuário e o histórico.
Como a auditoria registra tudo automaticamente?
Porque o registro está embutido na camada de dados: cada função de criar/editar/excluir chama logActivity. Como tudo passa por aqui, nada escapa.
Quais as limitações dessa abordagem?
Dados presos a um navegador, sem sincronização entre dispositivos e sem segurança real (senhas em texto). É adequado para um protótipo acadêmico, não para produção.
